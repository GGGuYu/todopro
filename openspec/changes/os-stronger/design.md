## Context

OpenSpec manages changes through a skill-driven workflow: `openspec-propose` creates artifacts, `openspec-apply-change` implements tasks, `openspec-archive-change` finalizes. All four OpenSpec skills are plain `SKILL.md` files in `.claude/skills/openspec-*/`, generated by `openspec init` from templates. They are overwritten on `openspec update`.

TodoPro's hook-based approach works but requires platform-specific installation. The insight: since OpenSpec skills are local files, we can **patch them in-place** to inject a review step. This avoids hooks entirely — the agent reads the patched skill instructions and follows them voluntarily, exactly as it already follows OpenSpec's own instructions.

## Goals / Non-Goals

**Goals:**
- Add an independent subagent review step between "all tasks done" and "archive" in the OpenSpec apply workflow
- Work purely through SKILL.md patches + one review guide file — no hooks, no external services, no npm dependencies
- Main agent never reads the review guide (avoids context bloat); only passes file paths to the subagent
- Max 2 review cycles with automatic circuit breaker
- Review findings are advisory, not mandatory — main agent judges truth + urgency
- `os-stronger init` is re-runnable and idempotent

**Non-Goals:**
- Replacing OpenSpec's task tracking or workflow
- Automatic file tracking (touched-files) — subagent uses git diff instead
- Loop-exit guard (four-exit nudge) — this is a hook-only feature, not achievable with pure prompts
- Supporting non-OpenSpec workflows

## Decisions

### Decision 1: Patch OpenSpec skills in-place (not create a separate skill)

**Choice**: `os-stronger init` reads the existing `.claude/skills/openspec-apply-change/SKILL.md` and `openspec-propose/SKILL.md`, modifies specific sections, and writes them back. Backups are made before patching.

**Why**: A separate companion skill requires the agent to remember to invoke it. Patching the skill the agent is already following guarantees the review step is reached — the agent sees it as part of the OpenSpec workflow, not an optional extra.

**Alternative considered**: A standalone `openspec-review` skill that the agent invokes after apply. Rejected because the agent might forget.

### Decision 2: Inject at `state: "all_done"` branch in apply-change

**Choice**: The patch replaces the "congratulate, suggest archive" text with a review workflow block. The existing "If state: all_done" check remains as the trigger.

**Why**: This is the precise moment when implementation is done but nothing has been archived yet. Inserting here means review happens at the right time with no additional triggering logic.

### Decision 3: Main agent passes paths, subagent reads content

**Choice**: The main agent checks `.todopro/review-guide.md` exists (boolean check), writes `.todopro/requirement-summary.md`, then tells the subagent: "Read `.todopro/review-guide.md` for rules, read `.todopro/requirement-summary.md` for what to check, read tasks.md, check git diff."

**Why**: The review guide can be long (multiple KB). If the main agent reads it, every turn's context window includes it — even turns before review. By keeping it as a path reference, the main agent's context stays clean. The subagent, which runs in a fresh context, reads everything it needs.

### Decision 4: Circuit breaker via tasks.md markers

**Choice**: Review fix tasks are named `Review N Fix - <description>`. The agent counts review cycles by scanning tasks.md for `Review \d+ Fix` patterns. Max 2.

**Why**: No external state file needed. The tasks.md is the single source of truth for work tracking. The pattern is human-readable and machine-parseable. Example:

```markdown
- [ ] Review 1 Fix - Missing error handling in auth module
- [ ] Review 1 Fix - Login page doesn't validate email
```

When the agent sees `Review 2 Fix` tasks, it knows this is the final review cycle. After fixing Review 2 items, archive regardless.

### Decision 5: Subagent findings are advisory

**Choice**: The review-guide.md instructs the subagent to use severity tiers (CRITICAL/ISSUE/SUGGEST) and explicitly states: "These are not mandatory. The main agent will judge whether each finding is actually true and worth fixing immediately." The main agent's patched apply-change instructions tell it to evaluate findings independently.

**Why**: Prevents the subagent from blocking progress on minor style issues. Gives the human user (via the main agent) final say on what gets fixed now vs. deferred. This mirrors TodoPro's existing review philosophy.

### Decision 6: Reuse existing review-subagent-prompt.md content

**Choice**: The `.todopro/review-guide.md` file is a near-copy of TodoPro's `skills/todopro/review-subagent-prompt.md`, with adjustments for the advisory-finding model and OpenSpec-specific file paths.

**Why**: The review rules (severity tiers, functional focus, anti-nitpicking guidance) are already battle-tested. No need to redesign.

### Decision 7: `os-stronger init` as standalone Node script

**Choice**: A single `os-stronger-init.js` file, zero dependencies, pure Node. It reads the target OpenSpec skill files, applies patches with simple string replacement, and creates `.todopro/review-guide.md`.

**Why**: Same zero-dependency philosophy as TodoPro. Easy to ship, easy to run, easy to understand.

## Risks / Trade-offs

- **OpenSpec update overwrites patches**: Users must re-run `os-stronger init` after `openspec update`. Mitigation: `os-stronger init` is fast and idempotent; document this clearly. Future: could hook into OpenSpec's update events.
- **String-based patching is fragile**: If OpenSpec significantly changes the skill text, patches may fail or produce garbled output. Mitigation: backup before patching; `os-stronger init --restore` flag to undo. Scope-limited — only 2 insertion points.
- **No automatic enforcement**: Without hooks, the agent might skip the review step if it ignores the patched instructions. Mitigation: OpenSpec's skill-following track record is strong; the same mechanism that makes `openspec-apply-change` work reliably will make the review step work.
- **Review cycle counting relies on text patterns**: If the agent formats fix tasks differently, the counter may miss them. Mitigation: the patched instructions include explicit formatting examples; the pattern `Review \d+ Fix` is simple enough to match even with minor variations.
